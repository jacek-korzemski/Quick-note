Tutorial: Aplikacja od zera

Backend Fat-Free PHP + Frontend React + Vite – notatki, użytkownicy, logowanie i tablice kanban (Jira)

1. Wstęp

W tym tutorialu zbudujesz od zera aplikację: notatki z rejestracją i logowaniem oraz tablice kanban (w stylu Jiry) z przeciąganiem zadań. Backend to API w JSON (Fat-Free Framework, PHP), frontend to SPA (React + Vite, TypeScript). Po przejściu wszystkich kroków będziesz miał działającą wersję z notatkami, użytkownikami i tablicami z zadaniami do przeciągania między kolumnami.

Wymagania

  • PHP 7.2+ (z rozszerzeniami: pdo_sqlite, json, mbstring)
  • Node.js (LTS) oraz npm lub pnpm
  • Przeglądarka i edytor kodu

2. Backend – postawienie od zera

Struktura katalogów

Utwórz w projekcie folder backend i wewnątrz:

  • backend/public/ – punkt wejścia HTTP (tu będzie index.php)
  • backend/app/ – konfiguracja i logika aplikacji
  • backend/app/Controllers/ – kontrolery
  • backend/app/Models/ – modele
  • backend/data/ – plik bazy SQLite (np. notes.db)

Fat-Free Framework

Pobierz rdzeń F3. Opcja bez Composera: skopiuj plik base.php z repozytorium fatfree-core do backend/ (tak aby ścieżka była backend/base.php). Opcja z Composerem:

cd backend
composer require bcosca/fatfree-core

Wtedy zamiast require '../base.php' użyjesz require '../vendor/autoload.php' i $f3 = \Base::instance();.

Plik wejścia public/index.php

Wszystkie żądania do API muszą trafiać do tego pliku. Na początku ustawiamy CORS (potrzebne, gdy frontend jest na innym porcie lub domenie), potem ładujemy framework, konfigurację, bazę i trasy.

<?php
// === CORS: pozwala przeglądarce na requesty z innej domeny/portu (np. front na :5173, API na :8080) ===
header('Access-Control-Allow-Origin: *');           // które origin mogą wywołać API
header('Access-Control-Allow-Methods: GET, POST, PUT, PATCH, DELETE, OPTIONS');
header('Access-Control-Allow-Headers: Content-Type, Authorization');  // np. Bearer token
header('Access-Control-Max-Age: 3600');              // cache odpowiedzi preflight (OPTIONS) na 1h
if ($_SERVER['REQUEST_METHOD'] === 'OPTIONS') {      // preflight: przeglądarka pyta „czy mogę wysłać POST z Authorization?”
    http_response_code(200);
    exit;                                            // nie uruchamiamy F3 – tylko nagłówki
}

// === Bootstrap Fat-Free: załaduj rdzeń i konfigurację ===
$f3 = require('../base.php');                        // zwraca instancję Base (singleton)
$f3->config('../app/config.ini');                   // DEBUG, AUTOLOAD, CORS itd.

// === Baza SQLite: jedna plikowa baza w data/notes.db ===
$db = new \DB\SQL('sqlite:' . realpath('../data') . '/notes.db');
$f3->set('DB', $db);                                 // kontrolery biorą $f3->get('DB')

// === Tworzenie tabel: IF NOT EXISTS – bezpieczne przy każdym uruchomieniu ===
$db->exec([
    "CREATE TABLE IF NOT EXISTS users (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        username TEXT NOT NULL UNIQUE,
        email TEXT NOT NULL UNIQUE,
        password TEXT NOT NULL,
        token TEXT,
        created_at TEXT DEFAULT (datetime('now')),
        updated_at TEXT DEFAULT (datetime('now'))
    )"
]);

$f3->set('CORS.origin', '*');

// === Obsługa OPTIONS dla dowolnej ścieżki (preflight dla /api/*) ===
$f3->route('OPTIONS /*', function($f3) {
    header('Access-Control-Allow-Origin: *');
    header('Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS');
    header('Access-Control-Allow-Headers: Content-Type, Authorization');
    header('Access-Control-Max-Age: 3600');
    $f3->status(200);
});

// Trasy auth dodamy w następnej sekcji
// $f3->route('POST /api/auth/register', 'AuthController->register');
// ...

$f3->run();                                          // dopasuj URL do trasy i wywołaj handler

Tip: Katalogiem dokumentu serwera musi być public, żeby nie udostępniać app/ ani data/ wprost. Przy wbudowanym serwerze PHP użyj -t public.

Konfiguracja app/config.ini

[globals]
DEBUG=3                                    ; poziom szczegółowości błędów (0–3); przy 500 zobaczysz stack trace
AUTOLOAD=../app/Controllers/;../app/Models/  ; ścieżki do autoloadu: F3 ładuje klasę z pliku o nazwie Klasa.php
CORS.origin=*                               ; wartość nagłówka Access-Control-Allow-Origin (F3 może go dodać)

AUTOLOAD – F3 ładuje klasy z tych katalogów (nazwa pliku = nazwa klasy, np. AuthController.php → AuthController).

Uruchomienie backendu

cd backend
php -S localhost:8080 -t public

Sprawdź w przeglądarce lub przez curl: http://localhost:8080/api/auth/me – na razie może zwrócić 404 (brak trasy), ale nie błąd PHP. Jeśli widzisz 404 z F3, routing działa.

Debugowanie (backend)

  • 404 na wszystkie /api/* – upewnij się, że żądanie trafia do index.php (np. -t public lub rewrite w Apache/Nginx).
  • CORS – w zakładce Network sprawdź, czy odpowiedź ma nagłówki Access-Control-Allow-*. OPTIONS musi kończyć się exit; bez wywoływania $f3->run().
  • Baza – folder data/ musi istnieć i być zapisywalny; po pierwszym uruchomieniu pojawi się notes.db.

3. Frontend – postawienie od zera

Projekt Vite + React + TypeScript

npm create vite@latest frontend -- --template react-ts
cd frontend
npm install
npm install react-router-dom

Proxy i alias w vite.config.ts

Żądania do /api mają być przekierowane na backend (np. port 8080). Dzięki temu w kodzie możesz używać względnych URL-i (/api/auth/login), a Vite w trybie dev przekieruje je na backend.

import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import path from 'path';

export default defineConfig({
  plugins: [react()],
  resolve: {
    // Dzięki temu w imporcie piszesz @/api/auth zamiast ../../../api/auth
    alias: {
      '@': path.resolve(__dirname, './src'),
    },
  },
  server: {
    // W dev: request do localhost:5173/api/... trafi na localhost:8080/api/...
    proxy: {
      '/api': {
        target: process.env.VITE_API_BASE ?? 'http://localhost:8080',
        changeOrigin: true,   // zmienia Host na target (potrzebne przy niektórych serwerach)
      },
    },
  },
});

Zmienna środowiskowa i config.ts

W developmentie zwykle nie ustawiasz VITE_API_BASE – wtedy API_BASE_URL jest pusty i requesty idą na ten sam host (np. http://localhost:5173/api/...), a Vite przekierowuje je na backend. W produkcji możesz ustawić pełny URL API.

// src/config.ts
// import.meta.env.VITE_* – zmienne z .env; w dev bez .env = '' → requesty idą na ten sam host (proxy)
export const API_BASE_URL = import.meta.env.VITE_API_BASE ?? '';

W requestach używaj: fetch(`${API_BASE_URL}/api/auth/login`, ...). Przy pustym API_BASE_URL będzie to /api/auth/login – czyli przez proxy.

Uruchomienie frontendu

npm run dev

Otwórz http://localhost:5173. Upewnij się, że backend działa na http://localhost:8080, żeby proxy działało.

Debugowanie (frontend)

  • Requesty do API kończą się 404 – sprawdź w Vite, czy proxy['/api'].target wskazuje na adres i port, na którym nasłuchuje backend.
  • CORS w dev przy proxy zwykle nie jest problemem – przeglądarka widzi ten sam origin (localhost:5173), a serwer Vite przekazuje request do backendu.

4. Model i kontroler użytkownika (backend)

Model app/Models/User.php

F3 używa mapperów do tabel. Klasa rozszerza \DB\SQL\Mapper, w konstruktorze podajesz bazę i nazwę tabeli. Metody load + dry() sprawdzają, czy znaleziono wiersz.

<?php
// Mapper F3: jedna klasa = jedna tabela; load() ładuje wiersz, dry() = true gdy brak wyniku
class User extends \DB\SQL\Mapper {

    public function __construct(\DB\SQL $db) {
        parent::__construct($db, 'users');   // powiązanie z tabelą 'users'
    }

    // Wyszukanie po emailu; load() ustawia wewnętrzny wskaźnik na wiersz (lub brak)
    public function findByEmail(string $email): bool {
        $this->load(['email = ?', $email]);
        return !$this->dry();                 // true = znaleziono (nie „pusty” wynik)
    }

    public function findByUsername(string $username): bool {
        $this->load(['username = ?', $username]);
        return !$this->dry();
    }

    // Używane przy każdym chronionym requestcie: token z nagłówka Authorization
    public function findByToken(string $token): bool {
        $this->load(['token = ?', $token]);
        return !$this->dry();
    }

    // Hasło nigdy w plain text – bcrypt (domyślny algorytm w PHP)
    public function setPassword(string $password): void {
        $this->password = password_hash($password, PASSWORD_BCRYPT);
    }

    public function verifyPassword(string $password): bool {
        return password_verify($password, $this->password);
    }

    // Token sesji: 64 znaki hex (32 bajty entropy); zapisujemy w modelu i w DB
    public function generateToken(): string {
        $token = bin2hex(random_bytes(32));
        $this->token = $token;
        return $token;
    }

    // Do odpowiedzi JSON – bez hasła i tokenu (bezpieczeństwo)
    public function toArray(): array {
        return [
            'id'         => $this->id,
            'username'   => $this->username,
            'email'      => $this->email,
            'created_at' => $this->created_at,
        ];
    }
}

Tip: W toArray() nie zwracaj hasła ani tokenu – tylko dane do wyświetlenia użytkownikowi.

Kontroler app/Controllers/AuthController.php

Wspólne metody: json (status + nagłówek JSON + echo), getBody (odczyt php://input, json_decode), getToken (nagłówek Authorization: Bearer <token>).

<?php
class AuthController {

    // Wysłanie odpowiedzi JSON: kod HTTP, nagłówek Content-Type, body jako JSON
    private function json($f3, $data, int $status = 200): void {
        $f3->status($status);
        header('Content-Type: application/json');
        header('Access-Control-Allow-Origin: *');
        echo json_encode($data);
    }

    // Body requestu (POST/PUT) – w PHP odczytujemy tylko raz z php://input
    private function getBody(): array {
        $body = file_get_contents('php://input');
        return json_decode($body, true) ?: [];
    }

    // Wyciągnięcie tokenu z nagłówka "Authorization: Bearer <token>"
    private function getToken($f3): ?string {
        $auth = $f3->get('HEADERS.Authorization') ?? $f3->get('HEADERS.authorization') ?? '';
        if (preg_match('/^Bearer\s+(.+)$/i', $auth, $matches)) {
            return $matches[1];
        }
        return null;
    }

    // Rejestracja: walidacja → sprawdzenie duplikatów → zapis użytkownika → zwrot user + token
    public function register(\Base $f3): void {
        $data = $this->getBody();
        $username = trim($data['username'] ?? '');
        $email    = trim($data['email'] ?? '');
        $password = $data['password'] ?? '';

        if (!$username || !$email || !$password) {
            $this->json($f3, ['error' => 'Wszystkie pola są wymagane.'], 400);
            return;
        }
        if (strlen($username) < 3) {
            $this->json($f3, ['error' => 'Nazwa użytkownika musi mieć min. 3 znaki.'], 400);
            return;
        }
        if (!filter_var($email, FILTER_VALIDATE_EMAIL)) {
            $this->json($f3, ['error' => 'Nieprawidłowy adres email.'], 400);
            return;
        }
        if (strlen($password) < 6) {
            $this->json($f3, ['error' => 'Hasło musi mieć min. 6 znaków.'], 400);
            return;
        }

        $user = new User($f3->get('DB'));
        if ($user->findByEmail($email)) {
            $this->json($f3, ['error' => 'Ten adres email jest już zajęty.'], 409);
            return;
        }
        if ($user->findByUsername($username)) {
            $this->json($f3, ['error' => 'Ta nazwa użytkownika jest już zajęta.'], 409);
            return;
        }

        $user->reset();
        $user->username = $username;
        $user->email = $email;
        $user->setPassword($password);
        $token = $user->generateToken();
        $user->save();

        $this->json($f3, ['user' => $user->toArray(), 'token' => $token], 201);
    }

    // Logowanie: znalezienie użytkownika po emailu, weryfikacja hasła, nowy token, zwrot user + token
    public function login(\Base $f3): void {
        $data = $this->getBody();
        $email    = trim($data['email'] ?? '');
        $password = $data['password'] ?? '';

        if (!$email || !$password) {
            $this->json($f3, ['error' => 'Email i hasło są wymagane.'], 400);
            return;
        }

        $user = new User($f3->get('DB'));
        if (!$user->findByEmail($email)) {
            $this->json($f3, ['error' => 'Nieprawidłowy email lub hasło.'], 401);
            return;
        }
        if (!$user->verifyPassword($password)) {
            $this->json($f3, ['error' => 'Nieprawidłowy email lub hasło.'], 401);
            return;
        }

        $token = $user->generateToken();
        $user->save();
        $this->json($f3, ['user' => $user->toArray(), 'token' => $token]);
    }

    // Wylogowanie: unieważnienie tokenu w bazie (front powinien usunąć token z localStorage)
    public function logout(\Base $f3): void {
        $token = $this->getToken($f3);
        if (!$token) {
            $this->json($f3, ['error' => 'Brak tokenu autoryzacji.'], 401);
            return;
        }
        $user = new User($f3->get('DB'));
        if ($user->findByToken($token)) {
            $user->token = null;
            $user->save();
        }
        $this->json($f3, ['message' => 'Wylogowano pomyślnie.']);
    }

    // Sprawdzenie sesji: token w nagłówku → znalezienie użytkownika → zwrot danych usera (np. po odświeżeniu strony)
    public function me(\Base $f3): void {
        $token = $this->getToken($f3);
        if (!$token) {
            $this->json($f3, ['error' => 'Brak tokenu autoryzacji.'], 401);
            return;
        }
        $user = new User($f3->get('DB'));
        if (!$user->findByToken($token)) {
            $this->json($f3, ['error' => 'Nieprawidłowy token.'], 401);
            return;
        }
        $this->json($f3, ['user' => $user->toArray()]);
    }
}

Trasy w public/index.php

Po fragmencie z $f3->route('OPTIONS /*', ...) dodaj:

$f3->route('POST /api/auth/login', 'AuthController->login');
$f3->route('POST /api/auth/logout', 'AuthController->logout');

Sprawdź: rejestracja POST /api/auth/register z body {"username":"test","email":"test@example.com","password":"haslo123"} powinna zwrócić 201 i obiekt z user oraz token.

Debugowanie

  • 500 – włącz DEBUG w config, sprawdź logi PHP; sprawdź składnię (np. typy w sygnaturach w starszym PHP).
  • 401 przy /me – w żądaniu musi być nagłówek Authorization: Bearer <token> (dokładnie taka forma).

5. Autentykacja po stronie frontendu

Warstwa API src/api/auth.ts

import { API_BASE_URL } from '@/config';

// Bazowy URL dla wszystkich endpointów auth (register, login, logout, me)
const API_BASE = `${API_BASE_URL}/api/auth`;

export interface User {
  id: number;
  username: string;
  email: string;
  created_at: string;
}

export interface AuthResponse {
  user: User;
  token: string;
}

export interface ApiError {
  error: string;
}

// Wspólna funkcja requestu: dopisuje token z localStorage do nagłówka Authorization (dla /me, logout)
async function request<T>(url: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('auth_token');
  const headers: Record<string, string> = {
    'Content-Type': 'application/json',
    ...(options.headers as Record<string, string>),
  };
  if (token) {
    headers['Authorization'] = `Bearer ${token}`;
  }
  const res = await fetch(`${API_BASE}${url}`, { ...options, headers });
  const data = await res.json();
  if (!res.ok) {
    throw new Error((data as ApiError).error || 'Wystąpił błąd serwera.');
  }
  return data as T;
}

export const authApi = {
  register(username: string, email: string, password: string) {
    return request<AuthResponse>('/register', {
      method: 'POST',
      body: JSON.stringify({ username, email, password }),
    });
  },
  login(email: string, password: string) {
    return request<AuthResponse>('/login', {
      method: 'POST',
      body: JSON.stringify({ email, password }),
    });
  },
  logout() {
    return request<{ message: string }>('/logout', { method: 'POST' });
  },
  me() {
    return request<{ user: User }>('/me');
  },
};

Kontekst src/context/AuthContext.tsx

import { createContext, useContext, useState, useEffect, useCallback, type ReactNode } from 'react';
import { authApi, type User } from '@/api/auth';

interface AuthContextType {
  user: User | null;
  loading: boolean;
  login: (email: string, password: string) => Promise<void>;
  register: (username: string, email: string, password: string) => Promise<void>;
  logout: () => Promise<void>;
}

const AuthContext = createContext<AuthContextType | null>(null);

export function AuthProvider({ children }: { children: ReactNode }) {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(true);

  // Przy starcie: jeśli jest token, wywołaj /me i ustaw usera; przy błędzie (np. 401) usuń token
  useEffect(() => {
    const token = localStorage.getItem('auth_token');
    if (!token) {
      setLoading(false);
      return;
    }
    authApi.me()
      .then(({ user }) => setUser(user))
      .catch(() => localStorage.removeItem('auth_token'))
      .finally(() => setLoading(false));
  }, []);

  const login = useCallback(async (email: string, password: string) => {
    const { user, token } = await authApi.login(email, password);
    localStorage.setItem('auth_token', token);  // zapis na późniejsze requesty i po odświeżeniu
    setUser(user);
  }, []);

  const register = useCallback(async (username: string, email: string, password: string) => {
    const { user, token } = await authApi.register(username, email, password);
    localStorage.setItem('auth_token', token);
    setUser(user);
  }, []);

  const logout = useCallback(async () => {
    try {
      await authApi.logout();   // backend zeruje token w DB
    } finally {
      localStorage.removeItem('auth_token');
      setUser(null);            // zawsze czyścimy stan, nawet gdy request się nie udał
    }
  }, []);

  return (
    <AuthContext.Provider value={{ user, loading, login, register, logout }}>
      {children}
    </AuthContext.Provider>
  );
}

// Hook do odczytu kontekstu – używaj w komponentach jako: const { user, login, logout } = useAuth();
export function useAuth() {
  const context = useContext(AuthContext);
  if (!context) throw new Error('useAuth must be used within an AuthProvider');
  return context;
}

Integracja w App.tsx

Owiń drzewo komponentów w AuthProvider. Na czas loading wyświetl loader (np. „Ładowanie…”), potem albo formularz logowania, albo treść dla zalogowanego użytkownika.

import { AuthProvider } from '@/context/AuthContext';

function App() {
  return (
    <AuthProvider>
      <YourContent />
    </AuthProvider>
  );
}

Prosty UI logowania

Komponent z formularzem: pola email, hasło; przy rejestracji dodatkowo username. Przyciski „Zaloguj” / „Zarejestruj” wywołują login lub register z useAuth(). W catch wyświetl error.message. Po udanym logowaniu stan user w kontekście będzie ustawiony – możesz pokazać np. „Witaj, {user.username}” i przycisk „Wyloguj” wywołujący logout().

Debugowanie

  • Failed to fetch – backend musi być włączony; sprawdź proxy w Vite.
  • 401 po odświeżeniu – token w localStorage jest wysyłany do /me; jeśli backend zwraca 401, token mógł być unieważniony – w catch usuwasz token i użytkownik wraca do ekranu logowania.

6. Kolejne endpointy i komponenty (Notatki)

Backend – tabela i model app/Models/Note.php

W index.php dodaj tworzenie tabeli (przed $f3->run()):

$db->exec([
    "CREATE TABLE IF NOT EXISTS notes (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        title TEXT NOT NULL,
        content TEXT,
        created_at TEXT NOT NULL,
        updated_at TEXT,
        user_id INTEGER NOT NULL,
        FOREIGN KEY (user_id) REFERENCES users(id)

]);

Model:

<?php
class Note extends \DB\SQL\Mapper {
    public function __construct(\DB\SQL $db) {
        parent::__construct($db, 'notes');
    }
    // Lista notatek użytkownika; find() zwraca wiele wierszy, order = najpierw ostatnio zmodyfikowane
    public function findByUser(int $userId): array {
        $results = $this->find(['user_id = ?', $userId], ['order' => 'updated_at DESC, created_at DESC']);
        return array_map(fn($n) => $n->toArray(), $results);
    }
    // Sprawdzenie, czy notatka o danym id należy do użytkownika (przy update/delete)
    public function findByIdAndUser(int $id, int $userId): bool {
        $this->load(['id = ? AND user_id = ?', $id, $userId]);
        return !$this->dry();
    }
    public function toArray(): array {
        return [
            'id' => (int) $this->id,
            'title' => $this->title,
            'content' => $this->content,
            'created_at' => $this->created_at,
            'updated_at' => $this->updated_at,
            'user_id' => (int) $this->user_id,
        ];
    }
}

Backend – app/Controllers/NoteController.php

W każdym działaniu: odczyt tokenu, znalezienie użytkownika po tokenie; brak użytkownika → 401. index – lista notatek użytkownika; create – walidacja title, zapis z user_id; update / delete – findByIdAndUser, potem aktualizacja lub erase(). Wzór: getCurrentUser($f3) zwraca User lub null.

<?php
class NoteController {
    // Wspólne: odpowiedź JSON, odczyt body, token z nagłówka, pobranie usera po tokenie
    private function json($f3, $data, int $status = 200): void {
        $f3->status($status);
        header('Content-Type: application/json');
        header('Access-Control-Allow-Origin: *');
        echo json_encode($data);
    }
    private function getBody(): array {
        $body = file_get_contents('php://input');
        return json_decode($body, true) ?: [];
    }
    private function getToken($f3): ?string {
        $auth = $f3->get('HEADERS.Authorization') ?? $f3->get('HEADERS.authorization') ?? '';
        if (preg_match('/^Bearer\s+(.+)$/i', $auth, $m)) return $m[1];
        return null;
    }
    private function getCurrentUser($f3): ?User {
        $token = $this->getToken($f3);
        if (!$token) return null;
        $user = new User($f3->get('DB'));
        return $user->findByToken($token) ? $user : null;
    }
    // GET /api/notes – lista notatek zalogowanego użytkownika
    public function index(\Base $f3): void {
        $user = $this->getCurrentUser($f3);
        if (!$user) { $this->json($f3, ['error' => 'Brak autoryzacji.'], 401); return; }
        $note = new Note($f3->get('DB'));
        $notes = $note->findByUser((int)$user->id);
        $this->json($f3, ['notes' => $notes]);
    }
    // POST /api/notes – nowa notatka (title wymagany, user_id z tokena)
    public function create(\Base $f3): void {
        $user = $this->getCurrentUser($f3);
        if (!$user) { $this->json($f3, ['error' => 'Brak autoryzacji.'], 401); return; }
        $data = $this->getBody();
        $title = trim($data['title'] ?? '');
        if (!$title) { $this->json($f3, ['error' => 'Tytuł jest wymagany.'], 400); return; }
        $note = new Note($f3->get('DB'));
        $note->reset();
        $note->title = $title;
        $note->content = trim($data['content'] ?? '');
        $note->user_id = (int)$user->id;
        $note->created_at = date('Y-m-d H:i:s');
        $note->updated_at = null;
        $note->save();
        $this->json($f3, ['note' => $note->toArray()], 201);
    }
    // PUT /api/notes/@id – aktualizacja tylko własnej notatki (findByIdAndUser)
    public function update(\Base $f3, array $params): void {
        $user = $this->getCurrentUser($f3);
        if (!$user) { $this->json($f3, ['error' => 'Brak autoryzacji.'], 401); return; }
        $id = (int)($params['id'] ?? 0);
        if (!$id) { $this->json($f3, ['error' => 'Nieprawidłowe ID.'], 400); return; }
        $note = new Note($f3->get('DB'));
        if (!$note->findByIdAndUser($id, (int)$user->id)) {
            $this->json($f3, ['error' => 'Notatka nie została znaleziona.'], 404); return;
        }
        $data = $this->getBody();
        $note->title = isset($data['title']) ? trim($data['title']) : $note->title;
        $note->content = isset($data['content']) ? trim($data['content']) : $note->content;
        $note->updated_at = date('Y-m-d H:i:s');
        $note->save();
        $this->json($f3, ['note' => $note->toArray()]);
    }
    // DELETE /api/notes/@id – usunięcie tylko własnej notatki
    public function delete(\Base $f3, array $params): void {
        $user = $this->getCurrentUser($f3);
        if (!$user) { $this->json($f3, ['error' => 'Brak autoryzacji.'], 401); return; }
        $id = (int)($params['id'] ?? 0);
        if (!$id) { $this->json($f3, ['error' => 'Nieprawidłowe ID.'], 400); return; }
        $note = new Note($f3->get('DB'));
        if (!$note->findByIdAndUser($id, (int)$user->id)) {
            $this->json($f3, ['error' => 'Notatka nie została znaleziona.'], 404); return;
        }
        $note->erase();
        $this->json($f3, ['message' => 'Notatka została usunięta.']);
    }
}

Backend – model app/Models/Task.php

<?php
// Status = kolumna kanban; position = kolejność w kolumnie (do reorder)
class Task extends \DB\SQL\Mapper {
    public const STATUS_TODO = 'todo';
    public const STATUS_IN_PROGRESS = 'in_progress';
    public const STATUS_VERIFICATION = 'verification';
    public const STATUS_DONE = 'done';
    public const STATUSES = [self::STATUS_TODO, self::STATUS_IN_PROGRESS, self::STATUS_VERIFICATION, self::STATUS_DONE];

    public function __construct(\DB\SQL $db) {
        parent::__construct($db, 'tasks');
    }
    // Lista zadań tablicy – tylko jeśli użytkownik ma dostęp do boardu (sprawdzenie w kontrolerze)
    public function findByBoard(int $boardId): array {
        $results = $this->find(
            ['board_id = ?', $boardId],
            ['order' => 'status ASC, position ASC, id ASC']
        );
        return array_map(fn($t) => $t->toArray(), $results);
    }
    public function findByIdAndBoard(int $id, int $boardId): bool {
        $this->load(['id = ? AND board_id = ?', $id, $boardId]);
        return !$this->dry();
    }
    // Następna wolna position w danej kolumnie (przy dodawaniu zadania lub przeniesieniu)
    public function getMaxPosition(int $boardId, string $status): int {
        $r = $this->db->exec(
            'SELECT COALESCE(MAX(position), -1) + 1 AS next_pos FROM tasks WHERE board_id = ? AND status = ?',
            [$boardId, $status]
        );
        return isset($r[0]['next_pos']) ? (int) $r[0]['next_pos'] : 0;
    }
    public function toArray(): array {
        return [
            'id' => (int) $this->id,
            'board_id' => (int) $this->board_id,
            'title' => $this->title,
            'content' => $this->content ?? '',
            'difficulty' => (int) ($this->difficulty ?? 0),
            'created_at' => $this->created_at,
            'status' => $this->status ?? self::STATUS_TODO,
            'position' => (int) ($this->position ?? 0),
        ];
    }
}

Backend – kontrolery BoardCategory, Board, Task i trasy

BoardCategoryController: index (GET /api/board-categories) – lista kategorii użytkownika; create, update, delete z sprawdzeniem getCurrentUser i findByIdAndUser. BoardController: index (lista tablic, opcjonalnie po board_category_id), getOne (GET /api/boards/@id), create, update, delete – zawsze weryfikacja usera. TaskController: index (GET /api/boards/@boardId/tasks) – lista zadań tablicy (najpierw sprawdź, czy użytkownik ma dostęp do boardu); create – nowe zadanie ze statusem todo i position = getMaxPosition(boardId, 'todo'); update – edycja tytułu/treści/statusu (przy zmianie statusu ustaw position = getMaxPosition(..., nowy_status)); reorder (PUT /api/boards/@boardId/tasks/reorder) – body: { "status": "in_progress", "task_ids": [3, 1, 5] } – ustawia każde zadanie na podany status i position = indeks w tablicy; delete – usunięcie zadania.

Przykład trasy reorder w index.php:

$f3->route('PUT /api/boards/@boardId/tasks/reorder', 'TaskController->reorder');

W TaskController->reorder: pobierz boardId z $params['boardId'], sprawdź dostęp użytkownika do boardu, odczytaj z body status i task_ids (tablica id zadań). Dla każdego id: findByIdAndBoard, ustaw task->status = status, task->position = indeks w pętli, task->save(). Na końcu zwróć zaktualizowaną listę zadań (np. ponowne findByBoard).

Komponent widoku tablicy – DndContext i kolumny

Struktura: jedna strona/widok na jedną tablicę (np. route /board/:boardId). W komponencie:

  • Stan: board, tasks, loading. W useEffect (zależność boardId): boardsApi.getById(id) i tasksApi.getByBoard(id), zapisz wyniki w state.
  • Stałe kolumn: COLUMN_IDS = ['todo', 'in_progress', 'verification', 'done'] i COLUMN_TITLES (np. „Do zrobienia”, „W trakcie”, …).
  • Podziel tasks po status: tasksByStatus[status] – tablica zadań danej kolumny.
  • Render: nagłówek z nazwą tablicy, przycisk „Dodaj zadanie”; siatka kolumn (np. grid 4 kolumny). Każda kolumna to komponent DroppableColumn (obiekt docelowy drop). Wewnątrz kolumny – lista zadań jako DraggableTaskCard (obiekt przeciągany).

Hook useDroppable – kolumna jako „strefa upuszczenia”

// W DroppableColumn: id kolumny = status (np. 'in_progress')
const { isOver, setNodeRef } = useDroppable({ id: status, disabled: readOnly });
return (
  <div ref={setNodeRef} className={isOver ? 'column-over' : ''}>
    <h3>{title}</h3>
    {tasks.map((task) => (
      <DraggableTaskCard key={task.id} task={task} onClick={...} readOnly={readOnly} />
    ))}
  </div>
);

setNodeRef – przypisz do elementu DOM, który ma być strefą drop. isOver – true, gdy użytkownik przeciąga element nad tą strefą (możesz podświetlić kolumnę).

Hook useDraggable – karta zadania

// W DraggableTaskCard: id = 'task-' + task.id, data = { task }
const { attributes, listeners, setNodeRef, transform, isDragging } = useDraggable({
  id: `task-${task.id}`,
  data: { task },
  disabled: readOnly,
});
const style = transform ? { transform: CSS.Translate.toString(transform) } : undefined;
return (
  <div ref={setNodeRef} style={style} {...listeners} {...attributes}
       className={isDragging ? 'dragging' : ''}>
    <div>{task.title}</div>
  </div>
);

listeners i attributes – przypnij do elementu, który ma być „chwytem” (np. cała karta). transform – przesunięcie w trakcie przeciągania (CSS transform). isDragging – np. zmniejsz opacity karty podczas przeciągania.

Obsługa onDragEnd – zapis nowego statusu i kolejności

W DndContext przekaż onDragEnd={handleDragEnd}. W handleDragEnd(event): active = przeciągany element (np. active.id === 'task-5'), over = element nad którym upuszczono (kolumna lub inna karta).

  • Jeśli over.id jest jednym z COLUMN_IDS (np. in_progress) – zadanie przeniesiono do innej kolumny. Wywołaj tasksApi.reorder(boardId, over.id, [...aktualneIdWKolumnieOver, activeTaskId]). Backend ustawi wszystkim zadaniom w tej kolumnie nowy status i position; po odpowiedzi odśwież listę zadań (refreshTasks).
  • Jeśli over to inna karta (np. task-7) – zmiana kolejności w tej samej kolumnie. Oblicz nową kolejność id (wstaw activeTaskId przed/za overTaskId), wywołaj tasksApi.reorder(boardId, statusKolumny, newOrderIds), potem odśwież.
const handleDragEnd = (event: DragEndEvent) => {
  const { active, over } = event;
  if (!over || over.id === active.id) return;
  const activeTaskId = Number(String(active.id).replace(/^task-/, ''));
  const activeTask = tasks.find((t) => t.id === activeTaskId);
  if (!activeTask) return;
  if (COLUMN_IDS.includes(over.id as TaskStatus)) {
    const newStatus = over.id as TaskStatus;
    const currentIds = tasks.filter((t) => t.status === newStatus).map((t) => t.id);
    tasksApi.reorder(boardId, newStatus, [...currentIds, activeTaskId]).then(() => refreshTasks());
    return;
  }
  // else: over to inna karta – zmiana kolejności w kolumnie (oblicz newOrder, wywołaj reorder)
};

Żeby przeciąganie nie uruchamiało się przy każdym kliknięciu (np. w link), użyj sensora z minimalnym przesunięciem:

// W JSX: <DndContext sensors={sensors} onDragEnd={handleDragEnd}>

distance: 8 – dopiero po przesunięciu kursora o 8 px rozpoczyna się drag (klik nie uruchamia drag).

Debugowanie (kanban)

  • Zadanie nie przenosi się – sprawdź, czy over.id w onDragEnd jest poprawnie rozpoznawane (kolumna vs karta); czy reorder wysyła poprawne task_ids i status.
  • 401 na /api/boards – token w nagłówku; czy użytkownik jest zalogowany.
  • Kolumna nie przyjmuje dropu – upewnij się, że useDroppable ma id równy jednemu z COLUMN_IDS (string), a nie np. liczbie.

7. Łączenie z backendem – podsumowanie

  • Dev: frontend (np. localhost:5173), proxy /api → backend (localhost:8080). API_BASE_URL pusty, więc fetch('/api/...') idzie przez proxy.
  • Token: po login/register zapis w localStorage; każdy chroniony request dodaje nagłówek Authorization: Bearer <token>.
  • Błędy: sprawdzaj res.ok, odczytaj data.error z JSON i pokaż użytkownikowi; przy 401 możesz usunąć token i przekierować do logowania.
  • CORS: potrzebny przy wywołaniu API z innej domeny/portu niż backend; przy proxy w dev request z frontu idzie do Vite, więc CORS dla API nie jest wtedy wymagany po stronie przeglądarki.

8. Debugowanie i typowe błędy

Backend

  • 404 na /api/* – routing: katalog dokumentu musi być public, żeby URL trafiał do index.php (np. php -S localhost:8080 -t public).
  • 500 – w config ustaw DEBUG=3, sprawdź logi PHP; sprawdź składnię i typy (PHP 7.2+).
  • Baza – katalog data/ musi istnieć i być zapisywalny; po pierwszym uruchomieniu powstaje plik notes.db.
  • Class not found – AUTOLOAD w config.ini musi wskazywać na katalogi Controllers i Models; nazwa pliku = nazwa klasy (np. AuthController.php → AuthController).

Frontend

  • Requesty nie dochodzą – sprawdź proxy w Vite (target, changeOrigin) i czy backend nasłuchuje na podanym porcie.
  • 401 po zalogowaniu – format nagłówka Authorization: Bearer <token>; endpoint /me musi zwracać 200 dla poprawnego tokenu.
  • Niespójny stan użytkownika – jeden źródłowy stan w AuthContext; komponenty czytają user przez useAuth().

Wspólne

  • Content-Type: requesty POST/PUT z body – Content-Type: application/json; odpowiedzi z API – ten sam nagłówek.
  • Body: w JS body: JSON.stringify({ ... }); w PHP odczyt tylko z file_get_contents('php://input') (strumień można odczytać raz).
